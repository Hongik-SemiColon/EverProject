# EverProject — 심전도 측정 기기 데이터 기반 앱

홍익대학교 캡스톤 디자인 프로젝트. 기업에서 제공하는 심전도(ECG) 측정 데이터를 활용해
데스크탑 앱(exe 프로토타입)을 개발합니다.

---

## 프로젝트 개요

- **협력 방식**: 기업은 심전도 데이터(필터링 포함)를 제공하고
- **결과물**: Windows exe 프로토타입
- **기술 스택**: 미정 (결정 후 `app/README.md`에 기록)
- **기획 도구**: Figma(디자인), Affine(문서) — 링크는 `docs/planning/` 참고

---

## 폴더 구조

```
app/            exe 프로토타입 소스 코드
data/
  samples/      기업 제공 심전도 샘플 데이터 (소용량만)
docs/
  meetings/     회의록 (YYYY-MM-DD-제목.md)
  reports/      캡스톤 보고서, 제작과정 보고서
  planning/     기능 기획서, Figma·Affine 링크
```

## 협업 규칙

자세한 규칙은 **[CONTRIBUTING.md](CONTRIBUTING.md)** 를 참고하세요. 요약:

- 브랜치 체계: `main`(안정) ← `develop`(통합) ← 작업 브랜치
- 작업 브랜치 이름: `전략/#이슈번호-제목` (예: `feat/#5-ecg-chart`)
- 커밋 메시지: `태그: 설명` 형식, 한글 사용 가능 (`feat:`, `fix:`, `docs:` 등)

---

## 🔄 개발 프로세스 (Work Process)

1. **이슈 생성**: 기능을 개발하기 전 GitHub Issues에 작업 내용을 등록하고 이슈 번호를 할당받습니다.
   (이슈 종류: `Feature` / `Bug` / `Refactor` / `Chore` / `Docs` / `Task` — 템플릿 제공)
2. **브랜치 생성**:
   - 브랜치를 생성하기 전 반드시 `develop` 브랜치를 최신 상태로 pull 받습니다.
   - `develop` 브랜치로부터 `feat/#이슈번호-제목` 형식으로 브랜치를 생성합니다.
3. **작업 및 커밋**: 작업 단위별로 커밋 컨벤션에 맞춰 커밋을 진행합니다.
4. **Pull Request (PR)**:
   - 본인의 브랜치 작업을 마친 후, GitHub에서 `develop` 브랜치로 PR을 보냅니다.
   - PR 작성 시 템플릿의 자가 검증 항목을 모두 점검합니다.
   - AI 리뷰어 외에 **최소 1명 이상의 인간 리뷰어에게 승인(approve)** 을 받은 후 `develop`에 머지합니다.

## 🏗️ Branch Strategy

### 1. 주요 브랜치 운영

- **`main`**: 실제 서비스로 출시될 수 있는 안정된 상태의 코드만 관리합니다.
- **`develop`**: 다음 출시 버전을 개발하는 통합 브랜치입니다. 모든 기능 개발의 시작점이며, 항상 빌드 오류가 없는 상태를 유지해야 합니다.
- **`feat`**: 새로운 기능을 개발할 때 사용합니다. 작업이 완료되면 `develop` 브랜치에 Pull Request(PR)를 보냅니다.
- **`fix`**: 버그 및 예외 사항을 수정할 때 사용합니다.
- **`refac`**: 기능 변경 없이 코드의 가독성을 높이거나 구조를 개선할 때 사용합니다.
- **`chore`**: 패키지 매니저 설정, 빌드 스크립트 수정, `.gitignore` 등 소스 코드 로직과 무관한 환경 설정을 수정할 때 사용합니다.
- **`docs`**: 문서를 생성하거나 수정할 때 사용합니다. (README, API 명세서, 주석 등)
- **`task`**: 어느 태그로도 분류하기 어려운 작업이나 하위 태스크를 포함하는 상위 작업에 사용합니다. (사용을 최소화하고, 가능하면 다른 태그로 세분화하여 사용합니다.)


### 2. 브랜치 명명 규칙 (Naming Convention)

이슈 번호와 명확한 작업 명을 포함하여 구성합니다.

- **형식**: `전략/#이슈번호-제목`
- **예시**:
  - `feat/#5-user`
  - `fix/#43-funding-api`
  - `chore/#10-setup`
  - `docs/#15-readme`
  - `task/#2-db-design`
