# 협업 규칙 (Conventions)

EverProject 팀의 브랜치 전략, 커밋 컨벤션, 개발 프로세스를 정의합니다.

## 🏗️ 브랜치 전략 (Branch Strategy)

### 1. 주요 브랜치 운영

| 브랜치 | 용도 |
| ------ | ---- |
| `main` | 발표·보고서 제출 등 실제로 공개될 수 있는 **안정된 상태**의 코드만 관리합니다. |
| `develop` | 다음 버전을 개발하는 **통합 브랜치**입니다. 모든 기능 개발의 시작점이며, 항상 빌드 오류가 없는 상태를 유지해야 합니다. |

### 2. 작업 브랜치 종류

| 접두어 | 용도 |
| ------ | ---- |
| `feat` | 새로운 기능 개발 |
| `fix` | 버그 및 예외 사항 수정 |
| `refac` | 기능 변경 없이 코드 가독성·구조 개선 |
| `chore` | 빌드 스크립트, `.gitignore` 등 소스 코드 로직과 무관한 환경 설정 수정 |
| `docs` | 문서 생성·수정 (README, 회의록, 보고서, 주석 등) |
| `task` | 어느 태그로도 분류하기 어려운 작업 (사용 최소화, 가능하면 다른 태그로 세분화) |

### 3. 브랜치 명명 규칙 (Naming Convention)

이슈 번호와 명확한 작업 명을 포함하여 구성합니다.

- **형식**: `전략/#이슈번호-제목`
- **예시**:
  - `feat/#5-ecg-chart`
  - `fix/#43-data-parser`
  - `chore/#10-setup`
  - `docs/#15-july-report`

## 🔄 개발 프로세스 (Work Process)

1. **이슈 생성**: 작업 전 GitHub Issues에 작업 내용을 등록하고 이슈 번호를 할당받습니다.
2. **브랜치 생성**:
   - 브랜치를 생성하기 전 반드시 `develop` 브랜치를 최신 상태로 pull 받습니다.
   - `develop`으로부터 `전략/#이슈번호-제목` 형식으로 브랜치를 생성합니다.
   ```
   git switch develop
   git pull origin develop
   git switch -c feat/#5-ecg-chart
   ```
3. **작업 및 커밋**: 작업 단위별로 아래 커밋 컨벤션에 맞춰 커밋합니다.
4. **Pull Request (PR)**:
   - 작업을 마친 후 GitHub에서 `develop` 브랜치로 PR을 보냅니다.
   - PR 작성 시 템플릿의 자가 검증 항목을 모두 점검합니다.
   - AI 리뷰어 외에 **최소 1명 이상의 인간 리뷰어에게 승인(approve)** 을 받은 후 `develop`에 머지합니다.
5. **main 반영**: 보고서 제출·발표 등 마일스톤 시점에 `develop` → `main` PR을 생성하여 반영합니다.

## 💬 커밋 컨벤션 (Commit Message)

- **형식**: `태그: 설명`
- **예시**: `feat: 심전도 그래프 화면 추가`

| 태그 | 설명 |
| ---- | ---- |
| `feat` | 새로운 기능 추가 또는 기존 기능의 요구사항 수정 |
| `fix` | 버그 수정 |
| `refactor` | 기능 변화 없이 코드 가독성·구조 개선 |
| `chore` | 빌드 설정, 환경 설정 등 코드 이외의 작업 |
| `docs` | 문서 수정 (README, 회의록, 보고서, 주석 등) |
| `style` | 코드 로직 변경 없는 스타일 수정 (들여쓰기, 포맷팅 등) |
| `test` | 테스트 코드 추가 및 수정 |
| `build` | 빌드 관련 파일 수정 |
| `release` | 버전 릴리즈 및 배포 관련 작업 |

## 💻 개발 스타일 가이드

> ⚠️ 기술 스택 확정 후 이 섹션을 구체화합니다. (`app/README.md`의 후보 비교 참고)

스택과 무관하게 지키는 공통 원칙:

1. **폴더 구조**: 기능(도메인) 기준으로 폴더를 나눠 관련 코드를 한곳에 모읍니다.
2. **이름 짓기**: 변수·함수는 역할이 드러나는 이름을 사용하고, 축약어를 지양합니다.
3. **데이터 파일**: 저장소에는 소용량 샘플만 커밋하고, 원본은 `data/README.md`의 클라우드 링크로 관리합니다.
4. **빌드 산출물**: exe 등 빌드 결과물은 커밋하지 않고 GitHub Releases로 배포합니다.
5. **비밀 정보**: API 키·비밀번호는 커밋하지 않습니다.
